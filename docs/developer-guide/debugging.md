# Debugging

Guidance for diagnosing issues during ELIPS development and in production deployments.

## Build for Debugging

### Debug build

```bash
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
```

The Debug build type enables:
- No optimization (`-O0`)
- Debug symbols (`-g`)
- Assertions (`_GLIBCXX_ASSERTIONS` and standard C asserts)
- Full stack traces on exceptions

### RelWithDebInfo (default build type)

```bash
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=RelWithDebInfo
```

This is the default when `CMAKE_BUILD_TYPE` is not specified. It provides optimized code with debug symbols — useful for profiling while retaining debuggability.

## Sanitizers

### AddressSanitizer (ASan) + UndefinedBehaviorSanitizer (UBSan)

```bash
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug \
    -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined -fno-omit-frame-pointer"
cmake --build build -j
ASAN_OPTIONS=detect_leaks=1 ctest --test-dir build --output-on-failure
```

ASan detects:
- Heap buffer overflows and underflows
- Use-after-free and use-after-return
- Double-free and invalid free
- Memory leaks (with `detect_leaks=1`)

UBSan detects:
- Signed integer overflow
- Division by zero
- Null pointer dereference
- Misaligned access
- Invalid enum values
- Float-to-int overflow

This configuration runs in CI on every push to `main` (see `.github/workflows/test.yml`).

### ThreadSanitizer (TSan)

```bash
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug \
    -DCMAKE_CXX_FLAGS="-fsanitize=thread -fno-omit-frame-pointer"
cmake --build build -j
ctest --test-dir build -R "Concurrency"
```

Useful for detecting data races in concurrency tests.

## Common Issues

### LockConflict

**Symptom:** `elips::LockConflict` exception on `open()`.  
**Cause:** Another writer process (or the same process in a leaked handle) holds the exclusive file lock on the database directory.  
**Diagnosis:**

```bash
# Check if a lock file exists
ls -la /path/to/db/LOCK

# Use lsof to find what's holding it (macOS/Linux)
lsof /path/to/db/LOCK
```

**Resolution:**
- Ensure the previous `ElipsInstance` was properly `close()`d or destroyed.
- If the process holding the lock crashed, delete the `LOCK` file manually (the next `open()` will re-create it).
- ELIPS is single-writer by design. Only one writer process can own a database at a time.

### DimensionMismatch

**Symptom:** `elips::DimensionMismatch` exception on `place()`.  
**Cause:** The vector's dimension does not match the vault's configured dimension.  
**Example:**

```
"expected dimension 1536, got 768"
```

**Resolution:** Ensure the embedding model outputs vectors of the same dimensionality as the database configuration. When reopening a database, the persisted `IDENTITY` file is authoritative — passing a conflicting `dimension` to `open()` throws `ConfigError`.

### ConfigError

**Symptom:** `elips::ConfigError` exception on `open()`.  
**Cause:** Reopening an existing database with a `dimension`, `metric`, or `index` that conflicts with the persisted `IDENTITY` file.  
**Example:**

```
"database identity dimension mismatch"
```

**Resolution:** When reopening a database, omit the `Config` (or match the original exactly). Pass no `Config` to `open()` to read the identity from disk:

```cpp
auto db = elips::open("/path/to/db");  // reads IDENTITY from disk
```

### InvalidVector

**Symptom:** `elips::InvalidVector` exception on `place()`.  
**Cause:** The vector contains `NaN`, `Inf`, or `-Inf` values.  
**Example:**

```
"vector contains non-finite values"
```

**Resolution:** Sanitize vectors before insertion. Cosine vectors should be L2-normalized. Euclidean vectors should not contain non-finite values.

### StorageError

**Symptom:** `elips::StorageError` exception during checkpoint, close, or WAL operations.  
**Cause:** Disk full, permission denied, or filesystem corruption.  
**Diagnosis:**

```bash
# Check disk space
df -h /path/to/db

# Check file permissions
ls -la /path/to/db/

# Check if snapshot or WAL file is corrupted
hexdump -C /path/to/db/elips.snapshot | head -20
hexdump -C /path/to/db/wal.log | head -20
```

### NotFound

**Symptom:** `elips::NotFound` exception on `fetch()` or `erase()`.  
**Cause:** The requested record ID does not exist in the vault, or the vault itself does not exist.  
**Resolution:** Check that the RecordID was generated by ELIPS (not fabricated), and that the record hasn't been erased.

## WAL Corruption Recovery

ELIPS's WAL is designed for robustness. On `open()`, the recovery process:

1. Reads the last snapshot (`elips.snapshot`)
2. Replays each WAL record (`wal.log`), validating the CRC32C checksum
3. Stops at the first corrupt or truncated record

Corrupt tail bytes are **cleanly dropped** — the valid prefix of the WAL is fully recovered. This means a partial write during a crash (e.g., power loss mid-write) never corrupts previously committed data.

### Diagnosing WAL issues

```bash
# Verify WAL integrity by reopening
elips verify /path/to/db

# Manually inspect WAL header
# Magic byte for WAL records: 0xE1105E01
# Each record: magic(4B) | op(1B) | vault_name(str) | id(16B) | vector(optional) | crc32c(4B)
hexdump -C /path/to/db/wal.log | head -50

# Check snapshot magic
# Snapshot magic: 0xE1105E01
hexdump -C /path/to/db/elips.snapshot | head -5
```

### Forcing recovery

If you suspect the WAL is corrupt but `open()` succeeds (because corrupt tails are truncated), you can force a clean snapshot:

```bash
# Open, checkpoint (truncates WAL), close
elips checkpoint /path/to/db
```

This creates a fresh snapshot and truncates the WAL, effectively compacting the database.

## GoogleTest Debugging Flags

### Running a single failing test

```bash
# Via CTest
ctest --test-dir build --output-on-failure -R "ExactIndexEdgeTest.EmptyIndexSearchReturnsEmpty"

# Via gtest directly (faster for iterative debugging)
./build/elips_tests --gtest_filter='ExactIndexEdgeTest.EmptyIndexSearchReturnsEmpty'
```

### Useful gtest flags

```bash
# Repeat a test to catch flaky failures
./build/elips_tests --gtest_filter='HnswRecall*' --gtest_repeat=10

# Shuffle test order to detect order-dependencies
./build/elips_tests --gtest_shuffle --gtest_random_seed=0

# Stop on first failure
./build/elips_tests --gtest_break_on_failure

# Show full failure output
./build/elips_tests --gtest_print_time=1

# Run disabled tests
./build/elips_tests --gtest_also_run_disabled_tests

# List all tests without running
./build/elips_tests --gtest_list_tests
```

### Debugging a test with a debugger

```bash
# Build Debug, run under lldb (macOS)
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
lldb ./build/elips_tests
(lldb) b ExactIndexEdgeTest_EmptyIndexSearchReturnsEmpty_Test::TestBody
(lldb) run --gtest_filter='ExactIndexEdgeTest.EmptyIndexSearchReturnsEmpty'

# Or with gdb (Linux)
gdb --args ./build/elips_tests --gtest_filter='RecoveryTest.WalReplayRecoversUncheckpointedWrites'
```

## Metal Shader Debugging (Apple GPU)

When GPU tests fail or produce incorrect results on Metal:

### Enable Metal validation

```bash
# Enable Metal API validation layer
export MTL_DEBUG_LAYER=1
export MTL_DEBUG_LAYER_ERROR_MODE=nslog
export MTL_SHADER_VALIDATION=1
./build/elips_tests --gtest_filter='*Metal*:*Gpu*'
```

### Capture a Metal frame

```bash
# Enable GPU frame capture
export MTL_CAPTURE_ENABLED=1
./build/elips_tests --gtest_filter='GpuBackendIntegrationTest.brute_force_index_search'
```

Open the resulting `.gputrace` file in Xcode for shader debugging, buffer inspection, and performance analysis.

### Check GPU availability

```bash
# Run GPU device probe in isolation
./build/elips_tests --gtest_filter='GpuDeviceManagerTest.probe_returns_results_on_supported_system'
```

If this test fails, verify:
- The device has a Metal-compatible GPU (Apple Silicon or discrete AMD GPU)
- The build was configured with Metal enabled (`ELIPS_GPU_METAL=ON`, the default on Apple)

### Common GPU issues

| Issue | Symptom | Likely Cause |
|---|---|---|
| `GTEST_SKIP` on GPU tests | No GPU available | Metal/cuda not enabled or device not supported |
| Wrong distance values | `cosine_distance_matches_cpu` fails | Half-precision rounding, incorrect normalization |
| Memory allocation failure | `allocate_and_free_succeed` fails | Insufficient GPU memory pool size |
| Dynamic batcher hang | Test timeout | Batching thread synchronization issue |

## Debugging Python Bindings

### Import errors

```bash
# Verify the module is built
ls -la bindings/python/elips/_core.*.so

# Check the Python path
PYTHONPATH=bindings/python python3 -c "import elips; print(elips.__version__)"

# Check for missing symbols
PYTHONPATH=bindings/python python3 -c "import elips; print(dir(elips))"
```

### Segfaults in Python

```bash
# Run with faulthandler
python3 -X faulthandler tests/python/test_bindings.py

# Run under lldb
lldb -- python3 tests/python/test_bindings.py
```

### Python-specific ASan

```bash
# Build the extension with sanitizers
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug \
    -DCMAKE_CXX_FLAGS="-fsanitize=address -fno-omit-frame-pointer" \
    -DELIPS_BUILD_PYTHON=ON
cmake --build build --target elips_pymodule -j

# Run Python under ASan
ASAN_OPTIONS=detect_leaks=1 \
    DYLD_INSERT_LIBRARIES=/path/to/libclang_rt.asan_osx_dynamic.dylib \
    PYTHONPATH=bindings/python python3 tests/python/test_bindings.py
```

## Debugging the CLI

```bash
# Run CLI in verbose mode
elips --verbose info /path/to/db

# Run CLI with EQL and check error output
elips query /path/to/db --eql 'seek in docs nearest [0.1,0.2] top 5 yield' 2>&1

# Validate WAL integrity with the CLI
elips verify /path/to/db
```

## Profiling

```bash
# Build with RelWithDebInfo for profiling
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build build -j

# Profile with Instruments (macOS)
instruments -t "Time Profiler" ./build/elips_bench

# Profile with perf (Linux)
perf record ./build/elips_bench
perf report
```
